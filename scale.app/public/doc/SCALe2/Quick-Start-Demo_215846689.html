SCALe2 : Quick Start Demo

Created by Wiki Administrator, last modified by Ebonie I McNeil on Jan 05, 2018

The first stage of a SCALe audit entails going from source code to a listing of diagnostics in the SCALe web application. This demo audits the source code of the common UNIX utility dos2unix, version 7.2.2, which is publicly available here. We run two popular static analysis tools on this source and show how to import the results into the SCALe auditing application.

Demo Overview

The following diagram gives a high-level overview of the demo. The source code for an application will be analyzed with Coverity and Fortify tools. Output from the tools will be uploaded to the SCALe web application. You will use the web application to evaluate the diagnostics Consider changing this version of the figure, so it shows only 1 auditor - a quick-start demo is usually tried out by only 1 person., determining if diagnostics are true violations of CERT Secure Coding rules. The results of your audit evaluation(s) will be compiled into a report.

 

Prerequisites for Completing This Demonstration

  1. *nix operating system
  2. GCC, make
  3. Coverity and Fortify analysis tools, with valid licenses. (If you don't have access to Coverity and Fortify analysis binaries (with valid licenses for both), we provide sample output, so skip to Step 3.)
  4. The SCALe web application is installed and running on a host accessible to the demo machine. Instructions for installing the web application are available here. 

  5. Web browser

Note: If using the test dev VM, skip to Step 3 and verify the listed files are located within the /home/scale/scale.app/demo/ directory.

Step 1: Build the Source

The first step is to download the source and build it. Download the source as a tarball archive from   here.

# Download the source code (if your machine doesn't have access to the 
# web, download elsewhere and transfer to your analysis machine)
wget http://waterlan.home.xs4all.nl/dos2unix/dos2unix-7.2.2.tar.gz
# Extract the archive.
tar -xzf dos2unix-7.2.2.tar.gz 
# Enter the new directory
cd dos2unix-7.2.2/ 
# Build the source
make

If any issues occur with the build process (such as missing dependencies), they must be repaired before proceeding.

Reason for building the code: Many static analyzers, such as Coverity and Fortify, require access to a functioning build of the source. These analyzers rely on compiler processes invoked during the build (for example, gcc) to parse the source. Some tools process the source directly, but often with lower-quality results.

Step 2: Static Analysis

After the build is done, use the static analysis tools. For this demo, use Coverity and Fortify (or skip to Step 3).

2.1 Using the Fortify Source Analyzer

Ensure the $FORTIFY_HOME environment variable is defined and points to the root of the Fortify installation on your system. E.g., in your ~/.bash_profile you could add the following line:

export FORTIFY_HOME=/home/username/HP_Fortify_SCA_and_Apps_4.10/bin

Here is how a Fortify analysis of C-code is run from a terminal: 

# Clean the build.
make clean
# Run the sourceanalyzer over the make process 
sourceanalyzer -b d2u make
# Run a Fortify scan (may take some time)
sourceanalyzer -b d2u -scan -f ~/d2u.fpr
# Generate a report 
ReportGenerator -format xml -f ~/d2u_fortify.xml -source ~/d2u.fpr -template $FORTIFY_HOME/Core/config/reports/DeveloperWorkbook.xml

First, remove (clean) files created by previous builds. This ensures that all files will be recompiled and Fortify can analyze the full build process. Next, run the Fortify sourceanalyzer tool, specifying a unique project identifier to the "-b" option (d2u) and specifying the build command (make). After this run completes, invoke sourceanalyzer again, with the same project id but this time with the "-scan" parameter specified. This invocation of sourceanalyzer analyzes the information collected from the first command. Results are dumped into ~/d2u.fpr, a binary file recognized by other applications in the Fortify tool chain. Then, generate a human-readable, XML summary of the analysis, using the ReportGenerator command shown above.  This command, which is bundled with Fortify, converts the Fortify report file d2u.fpr into a simple XML format. Be sure to substitute the full path to the Fortify installation in place of $FORTIFY_HOME. The output of this command,  ~/d2u_fortify.xml in this example, is saved for later use.

The output at the time we ran this tool is d2u_fortify.xml.

2.2 Using Coverity Prevent

Run a Coverity analysis, as follows:

# Clean the build.
make clean
# Run cov-build over the make process 
cov-build --dir ./d2ucov make
# Run a Coverity scan
cov-analyze --dir ./d2ucov --aggressiveness-level medium --all
# Generate a report
cov-format-errors --dir tmp --json-output-v2 ~/d2u_coverity.json

Start by removing files created by previous builds. Then, run the cov-build command, specifying an output directory (in this example, ./d2ucov) and the build command (make). When that run completes, run the cov-analyze command outlined above on the output directory.  Finally, generate a summary of the analysis with the cov-format-errors command. The file output of that command, ~/d2u_coverity.json, is saved for later use.

The output at the time we ran this tool looked like this: d2u_coverity.json

But My cov-build Didn't Work!

Depending on the configuration of your system, you may see a warning message like the following, when you attempt to run cov-build:

[WARNING] No files were emitted. This may be due to a problem with your
configuration or because no files were actually compiled by your build 
command. Please make sure you have configured the compilers actually 
used in the compilation.

This warning is usually the result of Coverity not detecting the compiler process during the build. The make script may not use gcc by default. You can work around this problem in multiple ways. You could force the use of gcc by setting the CC environment variable, as follows:

make clean
cov-build --dir ./d2ucov make CC=gcc

Also, you can configure Coverity, using the cov-configure command, to detect whatever compiler is invoked by the makefile. For example, if /usr/bin/cc were being used, you could do the following:

cov-configure --comptype gcc --compiler /usr/bin/cc --config custom_config
make clean
cov-build --dir ./d2ucov/ --config custom_config make

Either of these techniques should resolve the issue with the cov-build command.

Step 3: Prepare the Auditing Application

You have outputs from two tools stored in two files: d2u_fortify.xml and d2u_coverity.json. You will upload these files to the SCALe web application so that auditors can evaluate the associated diagnostics. 

Before proceeding, you need three things: the source code in a ZIP file, plus the two output files. In the example below, since the source was distributed as a tarball (meaning, already compressed using tar into the file named dos2unix-7.2.2.tar.gz), you must first untar and then zip the source code:

wget http://waterlan.home.xs4all.nl/dos2unix/dos2unix-7.2.2.tar.gz 
tar -xzf dos2unix-7.2.2.tar.gz 
zip -r dos2unix-7.2.2.zip dos2unix-7.2.2/

If you were not able to successfully produce outputs from any of the tools above, you may download our outputs here:

  • d2u_fortify.xml
  • d2u_coverity.json

Next, navigate to the URL of your local installation of the SCALE web application, using your preferred browser, and log in with the appropriate credentials, obtained from your SCALe administrator.   You should now see a list of existing SCALe projects maintained by this application. Select the "New project" link, as shown below to the right of the pointer:


Next, specify a project name and a description, then select "Create Project":

 

The application will now move to a page where we can upload the source code and tool outputs.  On this new page, click the Browse... button next to the Archive containing src label, and select the ZIP file created above. Next, indicate that you wish to upload both Coverity and Fortify output for a C project by clicking the checkboxes next to fortify / c and coverity / c. Finally, upload the tool outputs for Coverity and Fortify by clicking the Browse... buttons associated with those respective tools:


Scroll to the bottom of the page, and click the Create database button. The diagnostics from the two tools will be loaded into a common database:

After the page reloads, you will see the results of the operation, similar to the following: 

If any errors occurred during the construction of the database, they will be noted in the far right column . If no errors occurred, then scroll to the bottom of the page and click Create project from database:

The diagnostics are now ready to audit.

Step 4: Extracting Data

When you are finished auditing the SCALe data, you can export it from the web app using the Export CSV option. 

This produces a .csv (comma-separated values) file that can then be imported into Excel.
